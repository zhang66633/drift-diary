## 0. 概述

### 0.1 技术栈总览（已定）

| 层 | 选型 | 理由 |
|---|---|---|
| 引擎路线 | 纯 Web 单页应用 | 文字冒险不需游戏引擎重型能力；CSS 排版对书页感最友好；部署即静态站 |
| 前端框架 | React 18 + Vite + TypeScript | 生态最大，组件化清晰，TS 类型对 flag/状态系统刚需 |
| 状态管理 | Zustand | 轻量，API 简洁，适合游戏状态，TS 友好 |
| 样式 | Tailwind CSS + 自定义 CSS | 原子化开发快，书页风主题定制方便 |
| 数据格式 | JSON | 标准，TS 类型友好，解析零依赖 |
| UI 风格 | 书页风（羊皮纸 + 衬线 + 墨水感） | 最契合回忆录体，沉浸感最强 |
| 存档 | localStorage | 纯前端，简单，demo 够用 |
| 插图 | AI 生成（重要场景） | 增强沉浸感，以"插页版画"形式呈现 |
| 部署 | 静态站（Vercel / GitHub Pages / 本地） | 零后端，跨平台 |

### 0.2 设计原则

1. **数据驱动**：场景/选项/后果全在 JSON，引擎只解析渲染。新增章节不改代码。
2. **高内聚低耦合**：分层架构，数据/逻辑/视图严格分离，组件间通过接口通信。
3. **类型安全**：全链路 TypeScript，状态/资源/flag/场景均有类型定义。
4. **可拓展性**：上岛后功能（日记存档、物品图鉴、制作树）留接口不堵死。
5. **沉浸优先**：UI 服务沉浸感，书页风 + 插图 + 回顾旁白层。

**分层规则**：
- `types/`：纯类型，无逻辑，被所有层依赖。
- `data/`：纯数据，无逻辑，被 engine 依赖。
- `engine/`：纯逻辑，无 UI，不依赖 React。可独立测试。
- `store/`：连接 engine 与 ui，持有运行时状态。
- `ui/`：纯视图，只读 store 与调用 engine 接口，不含业务逻辑。
- `styles/`：纯样式。

**依赖方向**：`ui → store → engine → types ← data`。**禁止反向依赖**。

## 2. 类型系统（types/）

类型是整个系统的契约，先定义清楚。

### 2.1 状态类型（types/state.ts）

```typescript
// 玩家状态轴（5属性双轨）
export interface PlayerState {
  良心: number;      // 0-100，隐性·关系——道德自省，影响"他者"互动选项
  天意: number;      // 0-100，隐性·环境——已合并原"信念"，祈祷/虔诚/天佑
  勇气: number;      // 0-100，显性·行动——面对危险的胆量
  士气: number;      // 0-100，显性·生存——精神状态
  健康: number;      // 0-100，显性·生存——体力，≤0即死
}

// 资源轴
export interface Resources {
  钱: number;        // 镑/比索，内部统一用便士或分
  食物: number;      // 份
  淡水: number;      // 壶
  火药: number;      // 磅
  弹药: number;      // 颗
  工具: number;      // 件
  蜡: number;        // 份
  绳: number;        // 份
  装备: number;      // 件（枪/斧/艇等）
  墨水: number;      // 份（上岛后启用，demo 留 0）
  同伴: Companion[]; // 同伴列表
}

// 同伴
export interface Companion {
  id: string;        // 如 "suri"
  name: string;      // "苏里"
  loyalty: number;   // 0-100
}

// 技能
export type SkillId = '航海' | '木工' | '种植' | '狩猎' | '语言';
export type Skills = Record<SkillId, number>;  // 0-5 等级

// Flag（通用键值对）
export type Flags = Record<string, FlagValue>;
export type FlagValue = string | number | boolean | string[];

// 时间
export interface GameTime {
  date: string;        // ISO 日期，如 "1651-09-01"
  season: Season;      // 季节（影响健康）
}
export type Season = '旱季' | '雨季';

// 完整游戏状态
export interface GameState {
  state: PlayerState;
  resources: Resources;
  skills: Skills;
  flags: Flags;
  time: GameTime;
  currentSceneId: string;
  history: string[];              // 已访问场景 id 列表
  historySnapshots: SceneSnapshot[];  // 已访问场景的文本快照（用于回看）
}

// 场景快照（用于历史回看，只读）
export interface SceneSnapshot {
  sceneId: string;
  chapter: number;
  beat: string;
  date?: string;
  text: string[];                 // 当时渲染的正文（含条件性文本已解析）
  quotation?: string;
  illustrationCached?: string;
  timestamp: number;              // 访问时间
}
```

### 2.2 场景类型（types/scene.ts）

```typescript
// 场景
export interface Scene {
  id: string;                  // 如 "ch1_1_父训"
  chapter: number;             // 1, 2, 3...
  beat: string;                // 节拍名，如 "1.1 出身与父训"
  date?: string;               // 场景发生日期（用于时间推进）
  text: string[];              // 场景正文（分段，支持打字机效果）
  quotation?: string;          // 章首金句（仅章首场景）
  senses?: SenseTag[];         // 感官标签（调试/作者用）
  illustration?: IllustrationSpec;  // 插图（可选）
  choices?: Choice[];          // 单选选项（与 multiChoice 二选一）
  multiChoice?: MultiChoice;   // 多选选项（与 choices 二选一）
  narration?: NarrationSpec;   // 回顾旁白（可选，选择后触发）
  dream?: DreamSpec;           // 梦境片段（可选）
  ending?: EndingSpec;         // 结局标记（可选，触发结局浮层）
  providenceHook?: ProvidenceHook;  // 天意联动（可选）
  onEnter?: Consequence[];     // 进入场景即执行的后果（固定事件）
  chapterStart?: boolean;      // 章节开始场景（显示标题页+金句）
  chapterEnd?: boolean;        // 章节结束场景（触发自动存档+章节切换动效）
}

// 单选选项
export interface Choice {
  id: string;                  // 如 "父训_听"
  text: string;                // 选项显示文本
  condition?: Condition;       // 显示条件（如 flag 检查）——不满足则选项完全隐藏
  requirement?: Condition;     // 可用条件（如 勇气≥80 / 航海≥1）——不满足则选项灰色显示但可见，点击无效，显示requireFailText
  requireFailText?: string;    // requirement不满足时的提示文字（如"勇气不足"）
  consequences: Consequence[]; // 选择后果
  nextScene?: string;          // 跳转场景 id（可选，纯事件结局如return_menu可省略）
  narration?: NarrationSpec;   // 选择后回顾旁白（可选）
  death?: DeathSpec;           // 死亡标记（可选）
}

// 多选选项（如偷物资场景）
export interface MultiChoice {
  id: string;                  // 场景内唯一 id
  prompt: string;              // 提示语，如 "你打算从船上带走什么？"
  minSelect?: number;          // 最少选几项，默认 0
  maxSelect?: number;          // 最多选几项，默认全部
  options: MultiChoiceOption[];
  confirmText: string;         // 确认按钮文本，如 "出发"
  nextScene: string;           // 确认后跳转场景 id
  narration?: NarrationSpec;   // 确认后回顾旁白（可选）
}

export interface MultiChoiceOption {
  id: string;                  // 如 "窃_食"
  text: string;                // 选项显示文本
  consequences: Consequence[]; // 选中即执行的后果
  condition?: Condition;       // 显示条件
}

// 死亡标记
export interface DeathSpec {
  text: string;                // 死亡文本（主题化，如 "手稿在此处中断…"）
  reviveScene?: string;        // 回档场景 id（默认最近存档点）
}

// 后果（统一后果描述）
export interface Consequence {
  type: ConsequenceType;
  target?: string;             // 状态/资源/flag/技能 key
  value?: number | string | boolean | string[];
  operation?: 'set' | 'add' | 'push' | 'remove';  // 默认 'set'
  condition?: Condition;       // 满足条件才执行（可选）
}

export type ConsequenceType =
  | 'state'        // 改状态轴
  | 'resource'     // 改资源
  | 'skill'        // 改技能
  | 'flag'         // 改 flag
  | 'time'         // 推进时间
  | 'companion'    // 改同伴
  | 'providence'   // 改天意（隐藏）
  | 'event';       // 触发事件（如沉船固定事件）

// 条件
export interface Condition {
  type: 'flag' | 'state' | 'resource' | 'skill' | 'providence';
  target: string;
  operator: 'eq' | 'neq' | 'gt' | 'lt' | 'gte' | 'lte' | 'in' | 'not_in';
  value?: FlagValue | number;
}

// 条件性文本（场景正文/旁白按条件拼接）
export interface ConditionalText {
  segments: TextSegment[];     // 按顺序判断，满足条件的段拼接
}
export interface TextSegment {
  text: string;
  condition?: Condition;       // 无 condition 则恒显示
}

// 感官标签
export interface SenseTag {
  sense: '视' | '听' | '触' | '嗅' | '温';
  text: string;
}

// 回顾旁白
export interface NarrationSpec {
  text: string | ConditionalText;  // 支持条件性文本
  trigger: 'on_choice' | 'on_enter' | 'on_exit';
  condition?: Condition;       // 整体触发条件
}

// 梦境
export interface DreamSpec {
  text: string;
  trigger: 'on_choice' | 'on_enter';
}

// 结局
export interface EndingSpec {
  title: string;               // 结局名（四字短语，如"迷途知返"）
  text: string;                // 结局旁白（第一人称回忆录体，平淡但有遗憾）
  buttonText?: string;         // 关闭按钮文字，默认"合上书本"
}

// 天意联动
export interface ProvidenceHook {
  lowThreshold?: number;       // 天意低于此值触发
  highThreshold?: number;      // 天意高于此值触发
  effects?: Consequence[];     // 触发的后果（双向共用，向后兼容）
  lowEffects?: Consequence[];  // 仅低天意触发的后果
  highEffects?: Consequence[]; // 仅高天意触发的后果
}

// 插图
export interface IllustrationSpec {
  prompt: string;              // 生成 prompt（SDXL 风格）
  alt: string;                 // 替代文本
  size: IllustrationSize;
  position: 'top' | 'inline' | 'fullpage';
  cached?: string;             // 预生成缓存路径，如 "illustrations/ch1_1_父训.png"
}

export type IllustrationSize =
  | 'square_hd' | 'square' | 'portrait_4_3' | 'portrait_16_9'
  | 'landscape_4_3' | 'landscape_16_9';
```

## 4. 引擎层（engine/）

纯逻辑，无 UI，无 React 依赖。可独立单元测试。

### 4.1 SceneManager（场景管理器）

**职责**：加载场景数据、跳转场景、维护历史。

```typescript
interface SceneManager {
  loadChapter(chapter: number): Promise<Chapter>;
  getScene(sceneId: string): Scene | null;
  getCurrentScene(): Scene;
  goToScene(sceneId: string): void;
  getHistory(): string[];
  isSceneVisited(sceneId: string): boolean;
}
```

### 4.2 StateManager（状态管理器）

**职责**：读写玩家状态、资源、技能。所有数值修改经此。

```typescript
interface StateManager {
  getState(): PlayerState;
  getResources(): Resources;
  getSkills(): Skills;
  modifyState(target: keyof PlayerState, delta: number): void;
  modifyResource(target: keyof Resources, delta: number): void;
  modifySkill(target: SkillId, delta: number): void;
  addCompanion(companion: Companion): void;
  removeCompanion(id: string): void;
  clamp(value: number, min?: number, max?: number): number;  // 默认 0-100
}
```

### 4.3 FlagManager（Flag 管理器）

**职责**：读写 flag，支持条件判断。

```typescript
interface FlagManager {
  getFlag(key: string): FlagValue | undefined;
  setFlag(key: string, value: FlagValue): void;
  hasFlag(key: string): boolean;
  checkCondition(condition: Condition): boolean;
  checkConditions(conditions?: Condition[]): boolean;
}
```

### 4.4 TimeManager（时间管理器）

**职责**：日期推进、季节计算、9/1 主题锚检测。

```typescript
interface TimeManager {

  advanceTo(date: string): void;        // 推进到指定日期
```

### 4.5 ProvidenceEngine（天意引擎）

**职责**：管理隐藏天意值，触发厄运/恩典事件。

### 4.6 ConsequenceEngine（后果执行器）

**职责**：统一执行 Consequence[]，分发到各 Manager，处理条件性后果。

```typescript
interface ConsequenceEngine {
  execute(consequences: Consequence[]): void;
  // 内部分发（每个 Consequence 执行前先检查 condition，不满足则跳过）：
  // state → StateManager
  // resource → StateManager
  // skill → StateManager
  // flag → FlagManager
  // time → TimeManager
  // companion → StateManager
  // providence → ProvidenceEngine
  // event → 触发事件回调
}
```

**条件性后果执行**：
- 每个 Consequence 可带可选 `condition`
- 执行前调用 `FlagManager.checkCondition(consequence.condition)`，不满足则跳过该条
- 用于表达"若 X 则 Y"逻辑，如"若 `苏里_卖` 则士气-10"

**条件性文本解析**（辅助）：
- `ConditionalText` 由 ConsequenceEngine 的辅助方法 `resolveText(text: string | ConditionalText): string` 解析
- 遍历 segments，满足 condition 的拼接，不满足的跳过
- 用于场景正文/旁白按 flag 分化文本

### 4.7 SaveManager（存档管理器）

**职责**：localStorage 读写、多槽位、自动存档、日记存档（接口预留）。

```typescript
interface SaveManager {
  listSlots(): SaveSlot[];
  save(slotId: string, label: string, type?: 'auto' | 'manual' | 'diary'): void;
  load(slotId: string): GameState | null;
  deleteSlot(slotId: string): void;
  autoSave(label: string, gameState: GameState, sceneText: string): void;  // 每场景自动存档（进入非结局场景时触发）
  hasSave(): boolean;
  getLatestSave(): SaveSlot | null;      // 最近存档点（死亡回档用）
  migrate(oldSave: SaveData): SaveData;  // 版本迁移
}
```

**日记存档**（上岛后启用，接口预留）：
- `type: 'diary'` 槽位，每日记条目一个槽
- 受 `Resources.墨水` 限制：墨水≤0 时无法手动日记存档
- demo 不实现，但 `save()` 已支持 type 参数

## 5. 状态层（store/）

Zustand store，连接 engine 与 ui。

```typescript
interface GameStore {
  // 运行时状态
  gameState: GameState;
  currentScene: Scene;
  pendingNarration: NarrationSpec | null;   // 待显示的回顾旁白
  pendingDream: DreamSpec | null;
  pendingIllustration: IllustrationSpec | null;
  pendingDeath: DeathSpec | null;           // 待显示的死亡文本
  isTyping: boolean;                        // 打字机效果进行中
  isDebugMode: boolean;                     // 调试模式（快捷键 Ctrl+Shift+D 切换）
  historyView: SceneSnapshot | null;        // 历史回看中的场景（null=不在回看）
  
  // 动作
  init(): Promise<void>;
  chooseChoice(choiceId: string): void;     // 单选选择
  confirmMultiChoice(optionIds: string[]): void;  // 多选确认
  nextParagraph(): void;                    // 推进下一段文本
  skipTyping(): void;                       // 跳过打字机
  save(slotId: string): void;
  load(slotId: string): void;
  dismissNarration(): void;
  dismissDream(): void;
  dismissDeath(): void;                     // 死亡文本确认后回档
  toggleDebug(): void;                      // 切换调试模式
  openHistory(snapshot: SceneSnapshot): void;   // 打开历史回看
  closeHistory(): void;                     // 关闭历史回看
}
```

**关键设计**：`chooseChoice` 流程：
1. 找到 Choice
2. 检查 `choice.death`：若存在，设置 `pendingDeath`，等待玩家确认后回档（reviveScene 或最近存档点），流程终止
3. 执行 ConsequenceEngine.execute(choice.consequences)
4. 检查场景的 providenceHook，若触发则执行其 effects
5. 设置 pendingNarration / pendingDream（若有），等待玩家关闭后再跳转
6. 跳转至新场景（enterScene）：
   a. SceneManager.ensureSceneLoaded + goToScene
   b. 触发新场景的 onEnter 后果
   c. 推入 historySnapshots（场景文本快照）
   d. **若新场景非结局场景**，触发 SaveManager.autoSave（每场景自动存档）
   e. 检查健康≤0：若然，触发死亡（兜底机制）
   f. 检查 chapterEnd / chapterStart：触发章节标题页/章节切换动效

**`confirmMultiChoice` 流程**：
1. 校验 minSelect/maxSelect
2. 对每个选中 option 执行其 consequences
3. 检查场景 providenceHook
4. 设置 narration（若有）
5. goToScene(multiChoice.nextScene)
6. 同 chooseChoice 的 6–10 步

**死亡处理**：
- 触发条件：`choice.death` 标记（主要）或健康≤0（兜底）
- 显示：主题化文本"手稿在此处中断…"（DeathSpec.text）
- 确认后：回档到 `death.reviveScene` 或最近存档点，恢复该存档的 GameState
- 死亡不计入 historySnapshots（手稿中断，未写成）

**历史回看**：
- 入口：目录菜单（章节目录 + 已访问场景列表）
- 数据：从 `gameState.historySnapshots` 读取
- 交互：只读，不影响当前进度；关闭后回到当前场景
- UI：以"翻回前页"动效呈现，强化"手稿"隐喻

## 6. 视图层（ui/）

### 6.1 书页风设计规范

**核心元素**：
- **纸张**：羊皮纸/做旧纸纹理背景（CSS texture 或 SVG），微泛黄
- **字体**：正文衬线字体（如 "Noto Serif SC" / "Source Han Serif"），标题更庄重
- **墨水**：深褐墨色文字，非纯黑；重点字句略晕染
- **烛光**：页面边缘微暗角，模拟烛光照明
- **段落**：首行缩进，段间适度留白，行高宽松
- **选项**：以"墨水批注"或"页边注"形式呈现，悬停时墨水晕开
- **金句**：章首金句居中斜体，配装饰花纹
- **回顾旁白**：斜体 + 略浅墨色 + 左侧细线，区别于正文
- **梦境**：模糊滤镜 + 偏冷色调，区别于现实

**字体栈**：
```css
font-family: "Noto Serif SC", "Source Han Serif SC", "Songti SC", serif;
```

### 6.1.1 转场遮罩规范（Blocking Overlay）

为保证沉浸感与转场节奏，**任何全屏浮层（章节标题页、on_enter旁白、on_enter梦境、死亡页）显示时，正文区域不得提前加载文字**，遵循以下规范：

1. **进入新场景时序**：
   - 若新场景带 `chapterStart` / `narration.trigger: 'on_enter'` / `dream.trigger: 'on_enter'` / 健康≤0死亡：
     1. 正文区域立即淡出至15%透明度（仅保留暗化的纸张背景，不渲染正文、金句、选项）
     2. 浮层（章节标题/旁白/梦境/死亡）以淡入动画弹出
     3. **玩家点击"继续"关闭浮层后**，正文才开始打字机逐段显示
   - 若新场景无任何浮层：直接进入正文打字，无延迟

2. **on_choice 旁白/梦境**：保持现有逻辑——选择后立即弹出浮层，不提前切换场景，点"继续"后才跳转新场景

3. **视觉细节**：
   - 正文淡出时长：0.3s ease
   - 浮层淡入时长：0.2s ease
   - 遮罩背景：半透明深褐（rgba(26, 16, 8, 0.7)），不遮死，让玩家隐约感知"换了一页"
   - 状态栏保持可见（不属于正文区域，不淡出）

4. **状态逻辑**：
   - `isTyping` 初始值：`!hasBlockingOverlay`（有遮罩时不自动开始打字）
   - 浮层 dismiss 时：`dismissNarration/dismissDream/dismissChapterTitle` 统一设置 `isTyping: true` 启动正文
   - `hasBlockingEnterOverlay` 判断：`(pendingNarration?.trigger === 'on_enter') || (pendingDream?.trigger === 'on_enter') || chapterTitleVisible || pendingDeath`

## 7. 接口规范

### 7.1 组件间通信

**单向数据流**：
```
data (JSON) → engine (解析/执行) → store (运行时状态) → ui (渲染)
                ↑                                          │
                └──────── 玩家选择 (chooseChoice) ─────────┘
```

**事件回调**（engine → store，用于 UI 反馈）：
- `onSceneEnter(scene: Scene)`
- `onConsequenceExecuted(consequence: Consequence)`
- `onNarrationTriggered(narration: NarrationSpec)`
- `onDreamTriggered(dream: DreamSpec)`
- `onProvidenceHookTriggered(effects: Consequence[])`
- `onChapterChange(chapter: number)`
- `onSeptemberFirst()`  // 9/1 主题锚

### 7.2 引擎接口总表

| Manager | 关键方法 | 依赖 |
|---|---|---|
| SceneManager | loadChapter, getScene, goToScene | data/ |
| StateManager | modifyState, modifyResource, modifySkill, addCompanion | types/ |
| FlagManager | getFlag, setFlag, checkCondition | types/ |
| TimeManager | advanceTo, isSeptemberFirst | types/ |
| ProvidenceEngine | modify, checkHook | StateManager |
| ConsequenceEngine | execute | 上述所有 Manager |
| SaveManager | save, load, autoSave | localStorage, GameState |

**依赖规则**：
- ConsequenceEngine 依赖其他所有 Manager（分发器）
- 其他 Manager 互不依赖（高内聚低耦合）
- Store 依赖 ConsequenceEngine + SceneManager（编排）
- UI 只依赖 Store

## 8. 高内聚低耦合设计

### 8.1 分层架构

```
┌─────────────────────────────────────┐
│  UI 层 (ui/)                        │  React 组件，纯视图
│  BookShell, SceneRenderer, ...      │
├─────────────────────────────────────┤
│  Store 层 (store/)                  │  Zustand，编排引擎
│  gameStore                          │
├─────────────────────────────────────┤
│  Engine 层 (engine/)                │  纯逻辑，无 React
│  SceneManager, StateManager, ...    │
├─────────────────────────────────────┤
│  Data 层 (data/)                    │  纯 JSON 数据
│  ch1.json, ch2.json, ...            │
├─────────────────────────────────────┤
│  Types 层 (types/)                  │  纯类型，无逻辑
│  state.ts, scene.ts, save.ts        │
└─────────────────────────────────────┘
```

### 8.2 解耦手段

1. **接口隔离**：每个 Manager 有明确 interface，UI 只通过 store 调用，不直接碰 engine。
2. **数据驱动**：场景逻辑全在 JSON，引擎不含业务逻辑（不知道"父训"是什么，只执行 Consequence）。
3. **事件驱动**：engine 通过回调通知 store，store 决定 UI 反馈，engine 不关心 UI。
4. **类型契约**：所有跨层通信经 types/ 定义，编译期保证一致。
5. **纯函数优先**：Manager 方法尽量纯，副作用集中到 store。

### 8.3 测试策略

- **engine/**：纯逻辑，单元测试（Vitest）。每个 Manager 独立测试。
- **data/**：schema 校验测试（zod）。
- **ui/**：组件测试（Testing Library），mock store。
- **集成**：playtest 脚本，跑完 Ch1 验证状态/flag 流转。

## 11. 关键流程时序

### 11.1 玩家选择流程

```
玩家点击选项
    ↓
ui: ChoiceList.onClick(choiceId)  或  MultiChoiceList.onConfirm(optionIds)
    ↓
store: chooseChoice(choiceId)  或  confirmMultiChoice(optionIds)
    ↓
检查 choice.death（仅单选）
    ↓ (若 death 标记)
store: pendingDeath 设置 → ui: DeathLayer 显示
    ↓ (玩家确认)
engine: SaveManager.load(reviveScene 或 getLatestSave) → 流程终止
    ↓ (无 death)
engine: ConsequenceEngine.execute(consequences)
    ├→ 逐条检查 condition，不满足跳过
    ├→ StateManager.modifyState / modifyResource / modifySkill
    ├→ FlagManager.setFlag
    ├→ TimeManager.advanceTo
    ├→ ProvidenceEngine.modify
    └→ (event 回调)
    ↓
engine: ProvidenceEngine.checkHook(scene.providenceHook)
    ↓ (若触发)
store: pendingNarration / pendingDream 设置
    ↓
engine: SceneManager.goToScene(nextScene)
    ↓
engine: ConsequenceEngine.execute(newScene.onEnter)
    ↓
store: 推入 historySnapshots（场景文本快照）
    ↓
若新场景非结局场景 → engine: SaveManager.autoSave() （每场景自动存档）
    ↓
检查 健康≤0（兜底死亡）
    ↓ (若然)
store: pendingDeath 设置 → DeathLayer → 回档
    ↓ (否则)
ui: SceneRenderer 重新渲染（打字机效果）
    ↓ (若新场景有 illustration)
ui: IllustrationFrame 显示
    ↓ (若 newScene.chapterStart)
ui: 章节标题页浮层
    ↓ (若 newScene.chapterEnd)
ui: 章节切换动效（章末标记，供未来章节衔接用）
```

### 11.2 9/1 主题锚流程

```
TimeManager.advanceTo("1659-09-01")
    ↓
TimeManager.isSeptemberFirst() === true
    ↓
store: 触发 onSeptemberFirst 事件
    ↓
ui: DateBadge 高亮 + 回闪 1651/9/1 场景视觉
    ↓
ui: NarrationLayer 显示宿命旁白
```

| 复杂状态/flag 系统 | ✅ TS+Zustand | ⚠️ Python 可做但乱 | ✅ C# 强 | ❌ 弱 |
| 数据驱动可拓展 | ✅ JSON | ⚠️ 脚本混逻辑 | ✅ Ink | ❌ 节点混逻辑 |
| Web 部署跨平台 | ✅ 静态站 | ❌ 桌面应用 | ❌ 包体大 | ✅ 但单文件 |
| 插图系统集成 | ✅ img 标签 | ⚠️ 可但繁琐 | ✅ | ⚠️ |
| 后续上岛功能扩展 | ✅ 加数据+组件 | ⚠️ 重写 | ✅ | ❌ 架构限制 |

### A.2 为何选 Zustand 而非 Redux

- 文字冒险状态频繁小改（每次选择改 3–5 个值），Redux 样板过重
- Zustand API 简洁，TS 推断好，适合游戏状态
- 不需要 Redux DevTools 的时间旅行（游戏存档已覆盖）

### A.3 为何选 Tailwind 而非纯 CSS

- 书页风需要大量定制（纸张纹理、墨水晕染、烛光暗角），纯 CSS 可做但慢
- Tailwind 原子类快速搭建，自定义主题（颜色/字体/阴影）集中配置
- 与自定义 CSS 共存：复杂效果（纹理/滤镜）用自定义 CSS，布局用 Tailwind

### 13.1 内容设计硬约束

| 规则编号 | 规范内容 |
|---|---|
| R-001 | **游戏章节≠原著章节**：按戏剧弧分章，不按原书章节号切分。 |
| R-002 | **主线忠实原著**：核心情节节点不可删改——离家出走→首次风暴→被俘为奴→逃亡巴西→再次出海→沉船→荒岛→离岛。可加分支，但主干不歪。 |
| R-003 | **道德困境完整保留**：原著中灰色选择（如卖苏里为奴）不得洗白、不得跳过，必须给玩家选择权，并体现长期后果。 |
| R-004 | **Demo范围**：当前demo只实现登岛前两章（父训离家→雅茅斯风暴→几内亚航程引子），上岛后内容留接口不实现。 |
| R-005 | **文风要求**：第一人称回忆录体，译文语气匹配17世纪古典小说，每场景必须包含五感描写（视觉、听觉、触觉、嗅觉、温度）。 |
| R-006 | **场景密度要求**：不得过度压缩原著场景，每个重大事件必须保留足够叙事节拍——第一章不少于8拍，第二章不少于14拍。 |
| R-007 | **章节引言**：每章开头必须摘一句原著原文作为金句（`quotation`字段）。 |
| R-008 | **主题锚强化**：9月1日这个日期必须在游戏中反复出现并标记——既是第一次离家的日子，也是后来沉船的日子，强化宿命感。 |
| R-009 | **死亡判定**：两种触发——① 选项明确标记`death:true`（剧情杀）；② 健康值≤0（数值杀）。死亡页显示主题化文本，确认后回档至`reviveScene`或最近自动存档。 |
| R-010 | **章节边界**：用`chapterEnd:true`标记场景，进入时自动存档+章节转场动效；`chapterStart:true`显示章节标题页。 |
| R-011 | **叙事时序要求**：**严禁倒叙式选择**。场景文本必须截断在抉择点之前，玩家选择后通过`on_choice narration`展示选择结果，再进入下一场景。绝对禁止"文本已经把战斗/投降/选择结果写完了，然后再弹出选项问玩家选了什么"。正确结构：铺垫→抉择点→选项→结果旁白→下一场景。 |
| R-012 | **选项反馈强制要求**：**每个选项必须有反馈**，不能让玩家选了之后仿佛没选一样。最低要求是每个选项都写`on_choice`旁白（1-3句话即时描述后果）；重大分支（卖苏里、出海决定、道德困境）必须在后续场景用条件文本（`ConditionalText`）体现长期影响。数值变化玩家看不到，必须通过叙事反馈。 |
| R-013 | **禁止元叙事出戏文字&禁止UI标注路线**：选项文本中严禁出现"按原著XX"、"分支"、"原著路线"等打破第四面墙的文字；**UI上绝对禁止用任何视觉标记区分"原著路线"和"分支路线"**，所有选项视觉上完全平等，让玩家凭本心选择。`isCanon`字段仅用于数据层面标记（方便未来成就系统、路线统计、多周目对比），但**绝不可在游戏过程中向玩家暴露**。 |
| R-014 | **场景必须有出口**：每个有正文的场景必须有`choices`（玩家选择跳转）或`ending`标记（结局场景）或明确的自动跳转逻辑，不能写了一段正文没有任何选项导致玩家卡住。结局场景（带`ending`字段）不需要choices。 |
| R-015 | **Demo结束处理**：demo终章场景使用`{"type": "event", "target": "return_menu"}`事件后果，点击后清空状态返回主菜单，禁止nextScene指向自身造成死循环。 |
| R-016 | **结局系统**：分支结局通过`ending`标记触发，显示深色结局浮层——顶部是结局名（如"结局 · 迷途知返"），中间是结局旁白，底部"合上书本"按钮返回主菜单。结局场景不需要choices。结局基调遵循"平淡但有遗憾"原则——不走原著路线的安稳结局要有淡淡的怅惘感，不得写成完美无憾的大团圆。 |
| R-017 | **结局沉浸感**：结局浮层不显示任何"分支结局"、"Bad End/Good End"、"恭喜通关"等元叙事文字，不做任何路线标记，保持第一人称回忆录的沉浸感。结局标题用简洁的四字或短语（如"迷途知返"、"葬身鱼腹"、"荒岛余生"）。 |
| R-018 | **场景去重**：**同一个抉择点绝对不能拆成两个场景重复提问**。检查场景链时必须通读——如果场景A已经问了"做不做X"，场景B就不能再问"怎么做X/做多少X"（除非A只是铺垫没有真正的选择）。每个场景的choices必须直接回答该场景正文结尾抛出的问题，不能正文说"我要先除掉伊斯梅尔"，结果选项全是"带多少物资"（正文和选项割裂）。 |
| R-019 | **重大选择必须有长期叙事反馈**：数值变化玩家看不到，单纯加减良心/勇气点数不足以让玩家感受到选择的重量。道德困境/暴力选择（杀人、卖人、背叛）必须在**至少3个后续关键节点**通过`ConditionalText`条件旁白体现后果——可以是噩梦、回忆、愧疚感、他人态度变化，不能选完就当没发生过。 |
| R-020 | **选择的连锁后果可用条件选项实现**：利用`choice.condition`字段，可以让**同一场景同一位置的选项文字相同但互斥显示**，实现"之前的选择改变了当前可选项"。典型案例：选了"带最多物资"的玩家，在推摩尔人场景因被发现只能看到"杀人"一个选项，而其他玩家看到三个选项。禁止让选了X的玩家在Y场景看到与X矛盾的选项文字。 |
| R-021 | **连锁结局优于单选择死亡**：尽量避免"一个选错直接死"的设计，更有说服力的结局触发方式是**连续两个性格一致的选择叠加**——如"不敢猎狮+不敢接触黑人"=一味胆怯=饿死海上；"两次硬拼=鲁莽战死"。这种"性格即命运"的设计比随机死亡更公平，也更有主题深度。 |
| R-022 | **获得物/战利品后续一致性**：在场景中获得的物品（如狮肉、狮皮、豹皮、苏里的忠诚）必须在后续所有提到这些物品的场景用`ConditionalText`做条件区分——没猎到狮子的玩家不能有"狮肉吃完了/把狮肉递过去/船长买了狮皮"的叙述，否则造成严重叙事矛盾。 |
| R-023 | **灰色选项/状态门槛**：`Choice.requirement`字段用于实现"可见但不可选"的状态门槛选项。与`condition`（不满足则隐藏）不同，`requirement`不满足时选项**灰色显示但可见**（透明度40%、文字变灰、光标not-allowed），点击无效，`requireFailText`提示文字直接以红色小字显示在选项文字后方（如"（勇气不足）""需要航海术"）。典型用法：战死选项需要勇气≥80才能选择，勇气不足时选项可见但不可选，玩家能看到"战死"这个选项存在但无法选择——这比直接隐藏更有引导性，让玩家知道勇气高时有别的出路。 |
| R-024 | **技能门槛结局**：关键技能（如航海术）不足时应触发对应的无知结局而非卡死。例如：从摩尔人手中逃跑后，如果玩家没有学过航海术（航海技能<1），则无论选择"继续撑着"还是"冒险靠岸"都会在大海中迷失，触发"大海迷失"结局——这体现了"没有本事就不要逞强出海"的主题。技能门槛应通过**场景跳转条件**或**灰色选项**实现，而非直接死亡。 |
| R-025 | **复合条件支持**：`Condition`支持`and`/`or`复合条件字段，用于表达多flag组合判断（如"猎了狮 AND 杀了豹"）。格式：`{"and": [cond1, cond2, ...]}` 或 `{"or": [cond1, cond2, ...]}`，支持递归嵌套。当`and`/`or`存在时，`type`/`target`/`operator`/`value`叶子字段被忽略。典型用法：ch2_10船长买东西场景需要区分"有狮有豹/有狮无豹/无狮有豹/无狮无豹"四种组合，必须使用`and`复合条件才能精确匹配。 |
| R-026 | **ConditionalText互斥段规范**：`ConditionalText.segments`中，如果多个段是**同一内容的互斥变体**（如"独自到巴西"vs"苏里跟着上岸"），则每一段（包括默认段）**必须**有明确的`condition`，不能依赖"无condition=默认显示"。无condition的段语义是"永远显示的叠加段/收尾段"（通常以`\n\n`开头），会和所有满足条件的段拼接，若互斥变体缺少否定条件就会导致文本重复。修复方法：给默认段加上与其他变体互斥的否定条件（如`苏里≠留`）。 |

### 13.4 代码规范与避坑

以下为开发过程中实际踩过的坑，**所有后续开发必须规避**：

#### ❌ 坑001：JSON字符串引号错误
**现象**：场景JSON中中文文本内部误用ASCII双引号`"`代替中文直角引号`""`，导致JSON解析失败，章节无法加载，游戏卡在加载界面。
**规则**：所有中文文本内的引号必须用中文引号`""`或`''`，禁止在JSON字符串值中出现未转义的`"`。
**检查**：新增/修改JSON后必须用`JSON.parse`校验，或依赖TS类型+Vite构建检查。

#### ❌ 坑002：TypedText段落索引错位
**现象**：打字机组件第一段不显示、已打完段落消失、新段落跳出来覆盖旧段落。原因是：
- `displayedCount`（已完成段数）初始为0，但渲染循环`for (i < displayedCount)`导致索引0的第一段不渲染
- 当前正在打的段落内容错误地渲染在`displayedCount - 1`位置，造成索引错位
**正确逻辑**：
1. 循环渲染`0`到`finishedCount - 1`（已完成段，完整显示）
2. 在`finishedCount`位置渲染正在打的`currentText`
3. 不能用`finishedCount - 1`

#### ❌ 坑003：初始化死锁
**现象**：游戏永久卡在"手稿正在展开……"。原因：
- App.tsx中`initialized=false`时只渲染加载界面
- 但把`initialized=true`的`init()`函数放在BookShell组件的useEffect里调用
- BookShell恰恰需要`initialized=true`才会渲染，形成死循环
**规则**：全局初始化逻辑必须放在最外层App.tsx的useEffect中，不能依赖子组件挂载来触发。

#### ❌ 坑004：残留node进程占端口
**现象**：Vite启动时连续提示"Port 5173/5174/5175 is in use"，页面打开是旧版本代码。原因：多次启动dev server后旧进程未退出，残留node进程持续占用端口。
**解决流程**：
1. `netstat -ano | findstr ":517[0-9]"` 查出所有占用PID
2. `taskkill /F /PID <pid>` 逐个强制终止
3. 再`npm run dev`即可回到默认5173端口
**注意**：不要只杀最后一个看到的进程，必须枚举所有存活实例，防止漏杀。

#### ❌ 坑005：on_enter遮罩与正文同时渲染
**现象**：进入新场景时，旁白弹窗和正文打字同时进行，文字叠在一起视觉混乱。
**规则**：严格遵守R-204转场遮罩规范：
- `isTyping`初始值必须为`!hasBlockingOverlay`
- `hasBlockingOverlay`判断：`chapterTitleVisible || (pendingNarration?.trigger==='on_enter') || (pendingDream?.trigger==='on_enter') || pendingDeath || pendingEnding`
- 遮罩存在时BookShell不渲染TypedText/ChoiceList，只渲染暗化背景+浮层
- dismiss遮罩时统一设置`isTyping:true`启动打字

#### ❌ 坑006：Zustand store中重复action定义
**现象**：TypeScript报错"An object literal cannot have multiple properties with the same name"。原因：重构`dismissChapterTitle`后旧的定义未删除，重复出现在store对象中。
**规则**：修改store action时必须检查是否有旧定义残留。

#### ❌ 坑007：未使用变量导致TS编译失败
**现象**：`npx tsc --noEmit`报错"`isWaitingForNext` is declared but its value is never read"。
**规则**：删除未使用的局部变量，保持编译0警告0错误。

#### ❌ 坑008：Tooltip位置方向错误
**现象**：页面顶部状态栏的Tooltip向上弹出，被浏览器视口上边界截断看不到。
**规则**：靠近边缘的组件调用Tooltip时必须显式指定`position`：顶部元素用`position="bottom"`，底部元素用`position="top"`，不要依赖默认值。

#### ❌ 坑009：不要假设"Agent之前宣称修复了"就真的修了
**现象**：之前多次出现"已修复"但实际未验证的情况，导致问题反复。
**规则**：所有UI修改必须在浏览器中实际点击验证流程完整走完（至少走完第一章），不能只看代码逻辑正确就宣称完成。

#### ❌ 坑010：跨章节场景跳转未提前加载章节JSON
**现象**：从第一章结尾跳转到第二章时，控制台报错"Scene not found: ch2_1_几内亚"，游戏卡死。原因：SceneManager默认只加载了第一章，跳转其他章节时没有异步加载对应JSON。
**正确实现**：
1. SceneManager中添加`ensureSceneLoaded(sceneId)`异步方法，根据sceneId前缀（`ch(\d+)_`）解析目标章节，若未加载则先`loadChapter`
2. 所有场景跳转入口（`chooseChoice`、`dismissNarration`、`startNewGame`、`load`、`enterScene`）必须是`async`，在跳转前`await ensureSceneLoaded`
3. 异步修改后UI事件处理器不需要await（React事件处理器支持返回Promise），但内部逻辑必须等待加载完成

#### ❌ 坑011：ConsequenceEngine的companion类型简化写法支持
**现象**：`{"type": "companion", "target": "苏里", "value": true}`这种简化写法不生效，同伴无法正确添加/移除。原因：最初实现只支持完整Companion对象或remove操作，没有处理boolean值。
**规则**：
- `value: true` + `operation: "set"`：添加同伴（id/name用target，默认loyalty=100）
- `value: false`：移除同伴
- 完整Companion对象：按传入参数添加

#### ❌ 坑012：场景正文写完但漏写choices导致游戏卡住
**现象**：ch2_12巴西场景写完了正文，但没写choices数组，玩家看完正文后没有任何按钮可以点，游戏死锁。
**检查清单**：写完场景JSON后必须检查：
1. 如果是过场/叙述场景，必须有至少一个选项推进剧情（哪怕只有一个"日子一天天过去…"按钮）
2. 运行`node scripts/validate-scenes.mjs`验证所有跳转链路完整
3. 不要依赖on_enter旁白后自动跳转，玩家必须主动点击才能前进

#### ❌ 坑013：新增场景类型时忘记同步更新Zod schema
**现象**：添加`ending`结局场景后，Zod的SceneSchema仍然要求"必须有且仅有choices或multiChoice之一"，导致结局场景（无choices/multiChoice、只有ending字段）在校验时报错，游戏卡在加载界面。同样的问题：Choice.nextScene在TS类型中是optional（支持纯事件结局），但Zod schema中是required，也会报错。
**规则**：
1. 给Scene/Choice等数据类型新增字段时，必须**同时更新**：`src/types/scene.ts`（TS类型）和`src/engine/schema.ts`（Zod运行时校验）
2. 新增的"合法无choices场景"类型（如结局）必须在SceneSchema的refine校验中加白名单
3. 新增字段后必须同时跑`npx tsc --noEmit`和实际浏览器测试，Zod错误只在运行时抛出

#### ❌ 坑014：长文本全屏浮层被视口裁剪
**现象**：结局浮层EndingOverlay使用`flex items-center justify-center`垂直居中，当结局文本超过5段时总高度超出视口，导致顶部"— 结局 —"标记和标题被裁到视口上方，底部"合上书本"按钮被裁到视口下方，玩家只能看到中间正文段落，既不知道是结局也无法关闭。
**正确实现（双层flex布局）**：
```jsx
<div className="fixed inset-0 overflow-y-auto">           {/* 外层：可滚动 */}
  <div className="flex items-center justify-center min-h-full p-8">  {/* 中层：居中 */}
    <div className="...max-w-xl...">{/* 内层：内容 */}</div>
  </div>
</div>
```
- 外层`overflow-y-auto`允许内容超长时滚动
- 中层`min-h-full`保证短内容时垂直居中，长内容时被撑开
- 所有长文本浮层（结局、死亡、未来可能的长旁白）都必须使用此布局

#### ❌ 坑015：选项on_choice旁白与选择结果自相矛盾
**现象**："听船长的话，回家"这个选项最初的on_choice旁白写的是"如果我当时真的回了家，一切也许会不同。**但我没有**。羞耻心胜过了恐惧……"，然后跳转到第二章继续出海——旁白说"但我没有"回家，玩家明明选了回家却看到了说自己没回家的旁白，造成严重叙事矛盾。
**规则**：
1. 每个选项的on_choice旁白必须**忠实反映该选项的结果**，不能照搬其他选项的旁白
2. 分支选项（回家vs继续出海）的旁白必须分别写，不能复制粘贴
3. 写完选项后通读一遍：选A→看A的旁白→进入A的后续场景，叙事必须连贯，不能出现"选了A但旁白说我选了B"的矛盾

#### ❌ 坑016：chapterEnd标记位置错误 + 自动存档策略过于保守
**现象**：第二章的`chapterEnd: true`错误地标记在了`ch2_14_远航`（沉船场景），但玩家在该场景做出选择后还会继续进入`ch2_15_荒岛`（Demo真正终点），导致玩家玩到第二章终点时自动存档停留在沉船前，中途退出丢失大量进度；同时原设计"只在章末自动存档"策略过于保守，第二章流程较长，玩家中途退出时没有最近进度可恢复。
**规则**：
1. `chapterEnd: true`必须标记在**章节真正的最后一个场景**（玩家做完该场景所有选择后章节结束的位置），不能标记在中间场景
2. **自动存档策略已更新**：不再依赖`chapterEnd`触发，而是**每次进入非结局场景都自动存档**（见R-106），确保玩家在任何位置退出都能从最近场景继续
3. `chapterEnd`标记保留用于未来章节切换动效/过渡，不再承担自动存档触发职责
4. 结局场景（带`ending`字段）必须排除在自动存档之外，防止玩家在结局页返回主菜单后，"继续阅读"直接跳回结局造成死循环

#### ❌ 坑017：相邻场景重复同一抉择点 / 正文与选项割裂
**现象**：
- ch2_3（8月5日"船长让准备舢板"）选项是"偷偷搬东西/只准备鱼具"，ch2_4（8月20日"出海前一天"）又问"搬多少东西"，等于让玩家连续选两次"带多少物资"，重复且矛盾
- ch2_4正文最后一句是"必须先除掉伊斯梅尔"，把抉择点引向"如何处理摩尔人"，但三个选项全是关于物资准备的，正文抛出的问题和选项完全不对应，造成严重叙事割裂
- 苏里去留分支（ch2_11留苏里）后，ch2_12巴西和后续所有场景正文默认苏里已经不在，完全没有苏里留下的叙事痕迹
**规则**：
1. **写完场景链必须通读**：从玩家视角从头走一遍每个分支，检查：①上一个场景选的X，下一个场景是不是又让选X？②场景正文最后一句抛出的问题，选项是不是直接回答这个问题？③选了分支A的玩家，后续场景的正文/旁白是否和A一致？
2. 同一类抉择（物资准备、处理某人、去留）只能有**一个**抉择场景，不能拆成两个场景让玩家选两次
3. 同伴分支（带走/留下/卖掉/杀死）必须用ConditionalText在后续所有同伴应出现/不出现的场景给出对应旁白，不能默认所有人都走同一条路线
4. 场景正文结尾必须是"抉择点问句"（"我该怎么办？""带多少东西？""杀不杀他？"），选项必须是对这个问句的回答，禁止正文说东选项说西

#### ❌ 坑018：narration.condition字段从未被检查，导致旁白对所有玩家错误显示
**现象**：ch2_13种植园经营的on_enter旁白写了`condition: {"flag": "苏里", "op": "eq", "value": "卖"}`，原意是"只在卖了苏里时显示愧疚旁白"，但gameStore代码中设置pendingNarration时**完全没有读取和检查condition字段**，直接把整个narration对象塞给UI层，导致：①这段"想起苏里"的愧疚旁白对**所有玩家**（包括留苏里、送走苏里、谈判苏里的玩家）都显示；②ConditionalText也没有预解析，虽然UI层有resolveText兜底，但condition检查完全没做。
**正确实现（见R-110）**：
```typescript
// 设置pendingNarration前必须做两步检查：
const narrationCond = scene.narration.condition;
const narrationText = consequence.resolveText(scene.narration.text);
const narrationSatisfied = !narrationCond || flagMgr.checkCondition(narrationCond, ...);
if (narrationSatisfied && narrationText) {
  pendingNarration = { ...scene.narration, text: narrationText };
}
// 条件不满足或文本为空则不设置浮层，直接进入场景
```
此规则同样适用于`choice.narration`（on_choice旁白）和`multiChoice.narration`，三处都必须同样处理。

#### ❌ 坑019：正文text扩展为(string|ConditionalText)[]后，直接访问text[0]会导致类型错误
**现象**：将Scene.text类型从`string[]`扩展为`(string | ConditionalText)[]`以支持正文条件段落，导致`gameStore.save()`中直接将`scene.text[0]`传给saveMgr.save（期望string类型）产生TypeScript错误。
**正确处理**：所有读取`scene.text[n]`的地方必须用`consequence.resolveText()`包裹解析，例如：
```typescript
// 错误：
scene.text[0] ?? ''
// 正确：
consequence.resolveText(scene.text[0])
```
正文渲染层BookShell.tsx中已经通过`consequence.resolveText(t)`处理了每个段落，不需要额外修改。

#### ❌ 坑020：获得物/战利品在后续场景凭空出现
**现象**：ch2_7猎狮场景中，选了"不惊动狮子悄悄走"的玩家没有猎到狮子，但ch2_8黑人场景正文写"狮子肉已经吃完了"、交易旁白写"把一块狮肉递过去"，ch2_10葡船旁白写"船长出钱买了狮子皮"——等于没猎狮子的玩家凭空获得了狮子肉和狮子皮，严重破坏叙事一致性。
**规则**：在场景X中获得的物品（狮肉、狮皮、钱、装备等），后续所有场景Y的正文和旁白中提到该物品时，**必须**用ConditionalText按flag做条件区分，没获得的玩家看到的文本中不能出现该物品。每次写后续场景提到具体物品时，必须自问："所有玩家都有这个东西吗？如果不是，条件区分写了吗？"

#### ❌ 坑021：条件选项(choice.condition)的两个选项文字完全相同时，id必须不同
**现象**：实现"没猎狮+绕开→结局，猎了狮+绕开→继续"时，两个选项文字都是"绕开他们，不冒险接触"，但因为condition互斥显示，玩家只会看到一个。此时必须给两个选项**不同的id**（如`黑_绕`和`黑_绕_死`），否则Zod schema和React key都会出问题。
**规则**：条件互斥选项虽然文字相同，但id必须唯一，不能复用同一个id。

#### ❌ 坑022：结局场景(ending)必须有text字段（哪怕是空数组）
**现象**：新增"坐失良机"结局场景ch2_8_怯懦时，只写了`ending`字段但漏掉了`"text": []`，其他两个结局（迷途知返/宁死不屈）都有空text数组。由于SceneSchema中`text: z.array(...)`是必填字段（没有.optional()），Zod.parse在校验ch2.json时直接抛出异常，导致整个ch2无法加载——所有ch2的存档读取时都会崩溃，表现为"读档没反应/页面卡死"。
**规则**：结局场景虽然没有正文文本（内容在ending.text里），但**必须**显式写`"text": []`，不能省略text字段。

#### ❌ 坑023：SkillId类型必须与场景数据同步
**现象**：ch2_1场景中使用了`{"type": "skill", "target": "贸易", "value": 1}`，但SkillId类型定义中只有`'航海'|'木工'|'种植'|'狩猎'|'语言'`，没有"贸易"。ConsequenceEngine处理skill后果时调用`stateManager.upgradeSkill(target, value)`，而upgradeSkill内部有SKILL_META校验，遇到不在SKILL_META中的技能名会静默忽略或报错，导致该后果无效。
**规则**：新增任何技能名（如贸易、战斗、生存等）时，必须**同时更新**三处：①`src/types/state.ts`中的`SkillId`类型；②`StateManager.ts`中的`SKILL_META`对象；③`src/engine/schema.ts`中的Zod枚举。

#### ❌ 坑024：ConditionalText必须包含无condition的default segment（或覆盖所有分支）
**现象**：ch2_13的on_enter旁白有5个conditional segment（苏_卖/苏_谈/摩尔人_杀/战斗_战/战斗_降），但如果玩家选了"留苏里"且没杀摩尔人且战斗=降，则5个条件都不满足，resolveText返回空字符串，导致浮层显示空白（黑色背景上什么文字都没有）。
**规则**：所有ConditionalText必须有且仅有以下两种结构之一：①最后一个segment无condition（default兜底）；②所有分支的condition条件互斥且覆盖所有可能的玩家选择组合。否则会出现部分玩家看到空浮层。

#### ❌ 坑025：on_enter梦境和on_enter旁白不能同时触发
**现象**：ch2_14配置了on_enter的dream，同时日期是九月一日触发了自动on_enter旁白（"正是八年前的这个日子..."），两个浮层同时显示——梦境浮层z-50在上（深色半透明背景），旁白浮层z-40在下（浅色背景），梦境背景透明度0.85导致下方旁白框透上来，出现两个按钮（"醒来"+"继续"）和叠字。
**规则**：gameStore在enterScene中设置overlay时必须互斥检查：如果scene.dream存在且trigger为on_enter，则**禁止**同时设置pendingNarration（包括自动旁白）。时间锚点（如九月一日）的内容应直接整合进梦境文本，而非叠加显示。

### 13.5 内容审查工具

为防止人工疏漏导致的内容规范违反（如选项无反馈、flag缺乏长期反馈、获得物凭空出现等），项目内置自动化内容审查脚本`scripts/review-content.mjs`，每次内容扩写后必须运行检查。

#### 13.5.1 工具功能

```bash
node scripts/review-content.mjs
```

**审查项清单**：

| 审查项 | 对应规范 | 说明 |
|---|---|---|
| Flag使用统计 | R-019 | 统计每个flag在所有场景（正文/旁白/选项条件/后果）中的出现次数，不足3处发出警告，确保重大选择有足够长期叙事反馈 |
| 单选项无效果 | R-012 | 检测单选项场景是否有consequences/narration/dream/death等效果，无效果则警告 |
| 多选项同目标无条件 | R-012 | 检测所有选项跳转到同一目标且无条件区分的场景，可能违反"每个选项必须有反馈" |
| 梦境文本长度 | — | on_enter梦境超过300字警告，防止梦境承担过多剧情功能（梦境应为预感/象征，不应替代正文叙事） |
| 场景跳转完整性 | — | 验证所有choice.nextScene指向的场景都存在，避免跳转不存在的场景导致卡死 |
| 结局场景结构 | R-016 | 验证结局场景text为空数组，ending字段正确 |

#### 13.5.2 审查结果分级

- **❌ 错误（errors）**：必须立即修复，否则游戏无法运行或严重违反规范
- **⚠️ 警告（warnings）**：建议修复，可能影响叙事质量或违反设计原则
- **✅ 通过**：无错误无警告

#### 13.5.3 使用流程

1. **每次新增/修改场景后**必须运行`node scripts/review-content.mjs`
2. **错误必须全部修复**，警告需逐一评估：
   - 与设计原则冲突的警告必须修复
   - 确属合理设计例外的需在文档中说明理由
3. **新章节开发完成后**必须做全量审查，确保所有规范达标

#### 13.5.4 与现有验证脚本的关系

项目内置三类验证脚本，各司其职：

| 脚本 | 职责 | 运行时机 |
|---|---|---|
| `scripts/validate-schema.mjs` | Zod schema 数据格式校验 | 数据结构变更后 |
| `scripts/validate-scenes.mjs` | 场景跳转链路完整性检查 | 新增/删除场景后 |
| `scripts/review-content.mjs` | 内容规范审查（叙事质量/flag反馈/选项设计） | 每次内容修改后 |

### 13.6 开发流程约定

1. **先读再改**：修改任何文件前必须先Read获取最新内容，禁止凭记忆/历史缓存做Edit。
2. **类型检查**：每次代码修改后必须运行`npx tsc --noEmit`，确保0错误再继续。
3. **JSON校验**：场景JSON修改后必须确认解析正常，避免卡加载。
4. **热更新验证**：Vite热更新可能不刷新所有状态，修改store/转场逻辑后建议硬刷新（Ctrl+Shift+R）再测试。
5. **规范同步更新**：本规范为活文档，每踩一个新坑、每确立一个新硬约束，必须同步更新本章。
6. **可复用组件优先**：UI通用逻辑（Tooltip、弹窗、按钮等）抽独立组件，支持后续功能复用，不写内联一次性逻辑。
7. **内容审查**：每次新增/修改场景后必须运行`node scripts/review-content.mjs`，错误必须修复，警告需逐一评估。
